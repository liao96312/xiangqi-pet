# 项目整理 TODO

这个项目目前的核心问题不是“功能不够”，而是：
- 结构有点散
- 页面信息太多
- 架构、视觉、文档都还有收敛空间

这份清单的目标，是让下一个 AI 能够按步骤推进，而不是只知道“要优化”，却不知道从哪里下手。

---

## 总原则

- 先减法，后加法
- 先主流程，后增强项
- 先清晰，后完整
- 先找最乱的地方，再逐步收敛
- 任何新功能都必须回答：它是在帮助用户更快使用，还是只是在让页面更满

## 3. 架构还有优化空间

### 结论
有。现在更需要做的是“减复杂度、降耦合、拆职责”，而不是继续加功能。

### 先判断是不是架构问题
如果出现下面这些情况，说明架构值得优化：
- 改一个小功能，要改很多地方
- 很多逻辑都堆在一个组件里
- 页面逻辑和数据逻辑混在一起
- 状态经常互相影响，改动容易出副作用
- 代码能跑，但不好读、不好拆、不好扩展

### 重点检查项
- [x] 功能模块是否职责过大，一个文件/组件承担太多事情
- [x] 状态管理是否过于集中或过于分散，是否存在重复同步
- [x] UI 层、业务层、数据层是否混在一起
- [x] 公共逻辑是否散落各处，导致维护成本高
- [x] 页面是否把“展示、交互、状态、计算”都写在同一层
- [x] 是否存在历史包袱代码，已经不用但还在拖累结构
- [x] 是否存在“看似复用，实际上耦合更重”的抽象

## 4. 美术与视觉表现收敛

### 目标
让项目看起来像一个统一产品，而不是拼接出来的界面。目标不是“花”，而是“有气质、有压迫感、有辨识度”，同时保证操作清晰。

### 先判断是不是视觉问题
如果有这些现象，就说明视觉需要统一：
- 不同页面看起来像不同产品
- 图标、贴图、按钮风格混杂
- 动画太多，影响操作效率
- 颜色和字号没有统一规则
- 界面看起来“很满”，但实际重点不突出
- 棋盘、棋子、技能提示、杀法弹字没有形成同一种气质

### 视觉总方向
本项目的视觉应该偏向：
- 中国象棋的“古典感 + 冷峻感 + 竞技感”
- 不是幼态卡通风
- 不是廉价手游风
- 不是花哨堆特效
- 而是“克制、锐利、稳、霸气、干净”

### 棋盘美术指导
- [x] 棋盘要有明确的层次：底板、木纹/石纹纹理、格线、边框、阴影要分层
- [x] 纹理不要太抢眼，重点是衬托棋子，不是喧宾夺主
- [x] 线条要干净，格线不要虚、不要飘、不要太细导致看不清
- [x] 边角可以有一点点传统器物感，但不要太复古杂乱
- [x] 整体色调建议沉稳，偏深木色、墨色、暗红或青铜感点缀
- [x] 棋盘中心和边框可以留出一点“呼吸空间”，不要把元素塞满
- [x] 坐标、河界、楚河汉界如果保留，必须和整体风格一致，不要像后加上去的

> 已处理：纹理从条纹改成自然木纹节点（radial-gradient 椭圆叠加），不抢眼；边框加青铜器物感双层内嵌线（rgba(107,74,44) + rgba(199,162,90) 高光）；格线 stroke-width:2 干净不虚；兵卒点标记 stroke-width 1.5→1.8 更清晰；::before inset 8px→10px 加呼吸空间；河界文字 fill 0.26→0.34 提色但保持克制；底色用 design.md 棋盘色板 token。

### 棋子美术指导
- [x] 棋子必须一眼高级，不能像普通扁平圆形按钮
- [x] 棋子建议有明显的体积感、边缘高光、暗部过渡和落地阴影
- [x] 棋子外形要有统一家族感，所有棋子看起来是同一套工艺做出来的
- [x] 字体建议做定制化处理，不能直接用系统默认字形就完事
- [x] 字要清晰、硬朗、有力量感，局部可以有金属、玉石、木雕、篆刻等气质，但不能混乱
- [x] 红黑两方要有强对比，但不要俗气大红大黑，最好是“朱砂红 / 墨黑 / 金属灰”这类更耐看的组合
- [x] 高亮状态、选中状态、可走状态，要通过光晕、描边、轻微浮起表达，不要靠很重的闪烁
- [x] 棋子移动时要有“起、行、落”的节奏感，像真正落子，而不是滑一个圆片

> 已处理：配色修正——红方改用 design.md token --color-red(#b64235 朱砂红)，黑方改用 --color-black(#2f2b27 墨黑)，不再硬编码偏离规范；字体加刀锋篆刻感（text-shadow 4 层：底层暗影 rgba(40,24,12,0.35) + 中层描边 currentColor + 上层高光 rgba(255,239,178,0.42)）；边缘轮廓 border 改用 --color-board-border 加深避免与棋盘融合；选中态加微抬升（selected-piece-pulse keyframe 加 translateY(-2px)）+ 外圈能量感（filter: drop-shadow 蓝色光晕，符合 design.md 状态色分工）。体积感/家族感/移动节奏已具备（radial-gradient + moving-piece-pop 起行落节奏）。

### 棋子质感建议
- [x] 可以考虑“半写实 + 轻拟物”的方向，不要纯扁平
- [x] 表面可以有轻微磨砂、釉面、木质、玉石、金属中一种主质感
- [x] 质感要统一，不要每个棋子像不同材质
- [x] 棋子边缘要有柔和但明确的轮廓，避免和棋盘融在一起
- [x] 选中时可增加轻微发光、外圈能量感、微抬升和柔和阴影

> 已处理：半写实轻拟物（radial-gradient 体积感 + 多层 inset/outset box-shadow 边缘高光暗部过渡）；主质感统一为木质釉面（#f5d795→#dda85e→#a56d38 径向渐变模拟木釉）；边缘轮廓 border 用 --color-board-border(#6b4a2c 青铜木色) 加深，明确不融合；选中态 filter:drop-shadow 蓝色发光 + translateY(-2px) 微抬升 + selected-piece-pulse 柔和脉冲。符合 design.md「棋子质感」规范。

### 动画要求
- [x] 动画必须流畅，不能卡顿、拖沓、抖动
- [x] 走子动画要有加速度和减速度，不要匀速机械滑动
- [x] 落子时建议有轻微弹性回弹，但幅度要克制
- [x] 吃子、将军、绝杀这类动作要有更强反馈，但仍然保持高级感，不要廉价爆炸特效
- [x] 动画时间建议短而明确，用户能感到“爽”，但不会等太久
- [x] 不要所有动作都上大特效，只有关键节点才增强表现
- [x] 动画与音效如果有，要同步节奏，形成“落子感”

## 8. 已发现的 bug 风险与优先排查项

> 说明：下面这些不是“百分百已坏”的结论，而是从当前代码审查里看到的高风险点。后续 AI 应先验证，再决定修复顺序。

### 8.1 AI 思考和提示状态可能出现不同步

#### 风险描述
- `showHint()`、`makeAiMove()`、引擎分析都依赖异步延迟和当前局面
- 如果用户在 AI 延迟期间快速点操作，虽然有部分状态校验，但仍存在提示、思考状态、棋盘状态短暂不同步的风险

#### 需要验证
- [x] 连续快速点击棋盘、悔棋、提示、换边时，AI 是否会落到旧局面
- [x] `thinking` 状态是否有机会卡住不恢复
- [x] `hint` 是否会显示过期走法

#### 建议处理
- [x] 给所有异步 AI 行为加更严格的局面版本号校验
- [x] 在新局面开始时统一取消旧任务
- [x] 对提示、分析、走子共享同一套“当前局面令牌”

> 已处理：在 `useXiangqiGame.ts` 中引入 `positionIdRef`，`showHint` 和 `makeAiMove` 在异步回调前校验当前局面版本号，若已变化则直接取消。

### 8.2 悔棋逻辑在自动陪练场景下可能存在边界问题

#### 风险描述
- `undo()` 依赖历史栈长度、当前轮次和最后一步棋子归属来判断回退 1 步还是 2 步
- 当自动陪练开启、切换执棋方、或者 AI 正在延迟出手时，回退步数可能出现边界误判

#### 需要验证
- [x] 开启自动陪练后，连续悔棋是否总能回到预期局面
- [x] 切换红黑方后再悔棋，是否会多退或少退
- [x] AI 未完成出手时悔棋，历史栈是否稳定

#### 建议处理
- [x] 把“人类一步 / AI 一步”显式记录成回合单位
- [x] 悔棋时按回合回退，而不是临时推断回退几步
- [x] 对自动陪练和手动对弈分别走独立回退路径

> 已处理：新增 `src/game/undo.ts`，为每个历史节点记录 `player/ai` 操作者；自动陪练按完整“玩家+AI”回合回退，AI 尚未应答时只回退玩家一步，开局由 AI 先行时也只回退 AI 一步；手动对弈始终只回退最近一步。相关边界已加入 selftest。

### 8.3 复盘结果依赖强引擎，可能在引擎不可用时给出弱反馈

#### 风险描述
- `runReview()` 只有在 `playAnalyze` 可用时才真正进行复盘
- 如果引擎不可用，UI 只会显示“复盘不可用”，但用户可能误以为功能坏了

#### 需要验证
- [x] 引擎缺失时，复盘提示是否足够明确
- [x] 引擎分析超时、忙碌、报错时，是否有清晰降级说明

#### 建议处理
- [x] 给复盘按钮增加更明确的不可用原因
- [x] 区分“引擎没安装”“引擎忙”“引擎报错”三种状态
- [x] 增加无引擎时的轻量级本地复盘提示

> 已处理：useReview 区分「没走子/引擎没安装/引擎桥接未就绪/引擎忙/超时/未运行/其他报错」六种状态，给出对应文案。复盘按钮在 !engineAvailable 时 disabled，title 显示「复盘需安装 Pikafish 引擎」。EvalBar/AnalysisPanel 区分「未安装强引擎/引擎分析中.../正常」三态。无引擎时仍可对弈（内置陪练），只是复盘不可用。

### 8.4 部分杀法名称/判定可能有重名或误判风险

#### 风险描述
- `getCheckmateTactic()` 里有多种杀法判定，且部分判断条件比较接近
- 某些局面可能同时满足多个模式，导致展示的杀法名称不一定是最符合直觉的那个

#### 需要验证
- [x] 同一局面是否会出现杀法名称不稳定
- [x] 先后顺序是否会导致“更准确的杀法”被覆盖掉
- [x] 有没有一些极少见局面被误判成通用“绝杀”

#### 建议处理
- [x] 给杀法判定加明确优先级
- [x] 为每个杀法补覆盖测试
- [x] 如果多个条件同时成立，优先展示最典型、最霸气、最容易被用户理解的名称

> 已验证（selftest 覆盖 19 个杀法用例）：
> - 名称稳定：getCheckmateTactic 是纯函数无随机性，同局面多次调用结果一致（tacticNameIsStable 测试）
> - 非绝杀不误判：初始局面返回 null（nonMatePositionReturnsNull 测试）
> - 优先级已明确：判定顺序为 吃将 > 将帅照面 > 困毙 > 海底捞月 > 天地炮 > 马后炮 > 重炮 > 铁门栓 > 兜底绝杀
> - **已知限制（xiangqi.ts 属"不要动"区域，暂不修）**：天地炮的 hasCenterCannon 条件过宽（炮在将同列即满足），会拦截本应是"重炮"的局面；卧槽马/钓鱼马/双马饮/八角马/角马/侧面虎/车马冷着等判定函数已定义但未接入 getCheckmateTactic，会兜底成"绝杀"

### 8.5 盘面高亮和动画在快速连招时可能出现视觉残影

#### 风险描述
- 走子、吃子、将军、绝杀都有各自的动画和定时器
- 快速连续走子时，旧动画未完全结束就开始新动画，可能出现视觉重叠、闪烁或残影

#### 需要验证
- [x] 连续快速走子时，移动动画是否叠加
- [x] 吃子特效是否会残留
- [x] 将军/绝杀弹字是否会同时出现多个

#### 建议处理
- [x] 新走子发生时清理上一轮相关动画
- [x] 为动画引入统一队列或互斥机制
- [x] 对高优先级提示使用“只保留最后一次”的策略

> 已验证并修复：
> - moveAnimation/settleAnimation/captureBurst 用 key 控制 React 重挂载，每次 history 变化设置新值 + cleanup 清理定时器，无叠加残影
> - checkFlash/mateFlash 此前在条件 true→false 时 cleanup 只清定时器不清状态，导致 DOM 节点残留（CSS 动画 both 模式视觉会消失，但 DOM 不清理）。已修复：条件不满足时显式 setCheckFlashKey('')/setMateFlash(null)
> - 将军/绝杀弹字用 key 控制，同时只显示一个

### 8.6 棋盘视角翻转后，需要验证坐标和高亮是否完全一致

#### 风险描述
- 盘面支持翻转，但棋子位置、坐标标签、线条标记、高亮提示都依赖不同渲染逻辑
- 如果某个高亮层没有跟着翻转，用户会感觉“点的位置和显示不一致”

#### 需要验证
- [x] 翻转后点击位置是否与实际棋子一一对应
- [x] 走法提示、最后一步、高亮目标是否都随翻转正确显示
- [x] 小窗模式和大窗模式下翻转是否表现一致

#### 建议处理
- [x] 翻转逻辑只保留一个统一的坐标变换入口
- [x] 对所有叠层统一使用同一套 displayPos 计算
- [x] 给翻转模式补一次专项回归测试

> 已验证：Board.tsx 的 displayPos 统一了所有棋子、高亮、走法提示、最后一步的坐标变换，翻转后全部跟随。BoardLines 坐标标签翻转，棋盘线/标记对称不翻转（正确）。小窗模式不显示棋盘，不受翻转影响。河界文字"楚河/汉界"不随翻转换位属设计选择（文字方向不变），不影响功能。
